# 设计说明

本文件用于说明当前项目的实现边界、设计取舍与建议参数，
供 Codex 在实现时参考。

---

# 1. 总体思路

目标是实现一个轻量级的 screen-cam robust watermarking 原型：

- 显示端生成并嵌入鲁棒模板
- 手机拍屏后从照片中恢复嵌入信息
- 信息最小集为：
  - 固定设备标识
  - 粗粒度时间信息

这里不追求完整复现参考论文中的所有工程细节，而是保留其主骨架并做适度收缩。

---

# 2. 载荷设计

建议载荷格式：

`payload = device_id(32) || time_slot(16) || crc16(16)`

其中：

- `device_id = 0x1A2B3C4D`
- `time_slot = floor((unix_time - T0) / 300)`

说明：

- `T0` 为项目内部约定的固定起始时间
- `300` 秒表示 5 分钟时间粒度
- 5 分钟粒度足够用于取证
- `crc16` 用于检测恢复结果是否可信

当前阶段不使用 QR 编码。

---

# 3. 方法骨架

参考论文的主流程，可抽象为：

1. 生成消息模板 `Tm`
2. 生成同步模板 `Ta` 与 `Tb`
3. 将三者组合成分辨率自适应模板 `Tw`
4. 在待显示图像中按 JND 控制强度嵌入 `Tw`
5. 对拍摄图像做预处理，估计噪声分量
6. 基于同步模板完成定位与透视校正
7. 从校正后的消息区域在 DFT 域恢复 bitstream
8. 解析 payload 并做 CRC 校验

---

# 4. 当前建议保留的关键点

## 4.1 模板结构

尽量保留：

- `Tm`: message watermark template
- `Ta`: sync watermark template a
- `Tb`: sync watermark template b
- `Tw`: composed watermark template

## 4.2 分辨率自适应

尽量保留论文中的比例思路：

- `L1 = floor(0.47 * min(M, N))`
- `L2 = floor(0.06 * min(M, N))`

如果完整实现成本过高，也可以先做等比例近似实现，
但应在 README 中写明。

## 4.3 DFT 位置设计

优先参考论文中的参数：

- `R1 = {132, 145, 159, 172}` 用于 `Tm`
- `R2 = {20, 24, 28}` 用于 `Ta/Tb`

同步模板角度可参考：

- `theta_a = {125°, 130°, 135°, 140°, 145°}`
- `theta_b = {35°, 40°, 45°, 50°, 55°}`

如果第一版实现中需要简化，也可减少角度数或载波数量，
但要保留“不同类型同步模板”的思想。

---

# 5. 嵌入策略

建议嵌入公式：

`I_lum = bg + Tw * JND * alpha`

其中：

- `bg` 为亮度图
- `Tw` 为总模板
- `JND` 为简化的可感知阈值图
- `alpha` 为嵌入强度

## 初始参数建议

- `alpha = 0.35`

这是论文实验里表现较好的初值，可作为原型默认值，但不要求固定死。

建议支持命令行覆盖：

- `0.25`
- `0.30`
- `0.35`
- `0.40`

---

# 6. JND 实现建议

优先使用“简化 JND”，不要一开始做复杂视觉模型。

最低要求：

1. 将 RGB 转为亮度
2. 计算局部背景亮度
3. 计算局部梯度
4. 形成一个随内容变化的强度调制图

如果完整复现论文中的 Sobel + luminance masking + contrast masking 过重，
允许先做简化版，但命名中应避免误导。

例如：

- 若只做局部亮度和梯度调制，可写作 `simple_jnd_map`
- 不要声称“完全复现论文 JND”

---

# 7. 检测策略

第一版检测器不要求完全达到论文中的精度，但至少要包括：

1. 预处理
   - 从图像中提取亮度
   - 估计噪声分量
2. 基于同步模板定位
3. 透视校正
4. 从 `Tm` 对应区域提取 bitstream
5. 解析 payload
6. CRC 校验

建议逐步实现：

## 第一步
先支持从“数字攻击图像”恢复

## 第二步
再支持 perspective / JPEG / scaling 等

## 第三步
再尝试更像真实拍摄的失真模拟

---

# 8. 对论文的简化策略

为了适应硕士论文篇幅与工程能力限制，建议采用以下收缩策略：

1. 不做 QR 编码
2. 不做长消息
3. 不嵌入业务无关字段
4. 检测器第一版允许使用简化同步逻辑
5. 第一版以“可恢复”为目标，不追求最优 BER
6. 第一版以“单显示器”场景为主
7. 第一版可先做 Python 原型

---

# 9. 语言建议

## 第一阶段
优先 Python

原因：

- DFT / iDFT / 图像处理验证快
- 便于快速迭代参数
- 便于做 demo 与测试

## 第二阶段
实时嵌入模块迁移为 C

原因：

- Sway / wlroots 是 C 项目
- 最终 per-frame 嵌入应以 C 集成

## 可接受的组合

- `payload`：Python 原型，后续可迁移为 C
- `template_gen`：Python 原型
- `embed`：Python 原型，后续迁移为 C
- `detect`：可长期保留为 Python 离线工具
